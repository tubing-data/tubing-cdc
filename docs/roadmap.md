# Roadmap

Work toward DBLog-style behavior is grouped by dependency. Binlog position storage today is **not** a full replacement for the paper’s chunk-progress and HA state; expect new keys or a dedicated subsystem when P2–P5 land.

| Phase | Focus |
|-------|--------|
| **P0** | **Design** — Define a DBLog-aligned **event envelope** (action type, primary key, LSN or binlog position, origin = `log` \| `snapshot`, table identity, payload). Document compatibility or migration from the current JSON shape and `RowEventSink`. **Shipped:** types + `WithDBLogEnvelope` + [event-envelope.md](event-envelope.md). |
| **P1** | **Watermark infrastructure** — Create or configure the watermark table on the source; recognize watermark row updates in the binlog stream (map to canal events). **Shipped:** `WatermarkCreateTableSQL`, `Configs.Watermark` (auto-include table in canal filters), `WatermarkNotifier` + optional `ForwardWatermarkRowsToEventHandler`, `ParseWatermarkBinlogEvent`. |
| **P2** | **Chunked full state** — Primary-key-ordered chunks with configurable size; **persist chunk progress** (e.g. extend Badger/Redis with a separate namespace from binlog `PositionPersistence`, or another store). **Shipped:** `ChunkProgressRecord`, `BuildPKOrderedChunkSelect`, `Configs.ChunkProgressPersistence`, Badger-backed `ChunkProgressStore` (namespace separate from binlog keys; shared Badger when `BadgerDir` matches `PositionPersistence`), `TubingCDC.ChunkProgressStore`, `ReadChunkProgressFromBadger`. Runtime chunk execution + Algorithm 1 remain P3+. |
| **P3** | **Core algorithm** — Implement the paper’s watermark **Algorithm 1**: brief log pause for watermark writes + chunk read; resume; drop chunk PKs that appear between low and high watermark in the log; append surviving chunk rows in order. Align pause/resume with canal’s execution model and any ordered output buffer. **Shipped (core):** `Algorithm1Tracker` (`BeginCapture`, `OnWatermark`, `RecordTargetRowChange`, `ReconcileChunkRows`), `WatermarkUpdateValueSQL`, `ChainWatermarkNotifiers`, `Configs.Algorithm1` + `wrapHandlerWithAlgorithm1`, `TubingCDC.Canal()` for `Execute` during cycles. Operators still orchestrate pause/chunk `SELECT` and snapshot emission (P4 can add drivers and triggers). |
| **P4** | **Triggers and operations** — APIs or config to start full-state capture for all tables, one table, or selected PKs; pause and resume chunk processing. **Shipped:** `ChunkProcessingControl`, planning/queue/SQL helpers above, plus **built-in Algorithm 1 driver goroutine** `TubingCDC.StartAlgorithm1ChunkDriver` / `StopAlgorithm1ChunkDriver` (watermark `Execute`, chunk `SELECT`, `ReconcileChunkRows`, `RowSink.Emit`). Does **not** pause binlog consumption (see [algorithm1-chunk-driver.md](algorithm1-chunk-driver.md)). |
| **P5** | **High availability** — Leader election and standby processes (paper uses ZooKeeper; alternatives such as etcd or Redis are plausible). **Shipped:** Redis lease + renewal (`Configs.LeaderElection`, `AcquireRedisLeaderSession`, `RunTubingCDCWithLeaderElection` / `RunTubingCDCWithLeaderElectionFrom`; recreates `TubingCDC` per leader term because canal is not restartable after `Close`). |
| **P6** | **Multi-database (optional)** — **MySQL (shipped):** multiple independent MySQL sources via `MySQLSourceSpec`, `NewMultiMySQLCDC`, `MultiMySQLCDC.Run` / `Close`, and `ApplyMySQLSourcePersistenceScope` when sharing Badger/Redis. **TODO:** PostgreSQL (logical replication) and other engines; a shared `ReplicationSource` interface above canal. |
| **CI and Docker** | Expand docker-based workflows so the full CDC path (and, later, watermark/chunk paths) is easy to run end-to-end in CI or locally; extend integration tests as features ship. |

What exists today vs the paper: [coverage-vs-dblog.md](coverage-vs-dblog.md). Background: [context.md](context.md).
